# TeacherPilot: The AI Command Center for Educators

**Eliminating decision fatigue for overwhelmed teachers**

> "The AI did 100 things. You see 3."

---

## What is this?

TeacherPilot is a daily command center built for a real teacher managing 272 students across 13 groups. The core problem it solves: teachers drown in administrative noise — grading backlogs, parent emails, lesson prep, attendance flags — spread across multiple tools with no coherent picture of what actually matters *right now*.

Instead of yet another chat interface that requires constant prompt engineering, TeacherPilot surfaces exactly **3 prioritized actions per day**, generated by Mistral AI after analyzing the teacher's live schedule, pending tasks, and deadlines. The teacher sees only what needs to happen next, not a firehose of everything.

---

## Meet Marimba

Marimba is the AI companion at the heart of TeacherPilot — a fox character inspired by the founder's chocolate border collie, and named after the traditional Colombian instrument. Clever, loyal, gets things done.

Marimba is always present as a small avatar. The teacher can tap it or press a mic button to speak. Marimba responds in a consistent voice (ElevenLabs) and updates the screen accordingly. Responses are concise — never more than 2–3 sentences.

**Visual states:**

| State | Behavior |
|---|---|
| Idle | Subtle breathing / blinking animation (CSS) |
| Listening | Ears perk up, eyes widen, mic indicator visible |
| Thinking | Slight head tilt, animated dots or swirl |
| Speaking | Mouth open/close synced to audio volume |

**Example interactions:**
- *"What's my next class?"* → "You have 9A1 in the Codingspace at 11:30. Want me to prep it?"
- *"What's urgent?"* → "One thing: the coordinator needs a reply about Tuesday's event."
- *"Help me prep 6B2"* → Opens classroom card for 6B2 + reads briefing aloud

**Voice pipeline:**
```
Teacher speaks → Voxtral Realtime (transcription)
                        ↓
               Mistral Large 3 (reasoning)
                        ↓
             ElevenLabs (consistent voice ID)
                        ↓
              Marimba speaks + screen updates
```

The mic button is always accessible. Marimba's voice is always the same voice ID — not generic TTS, a real persona.

## Design philosophy

Two principles guide every decision in this project:

**Emotional connection over utility.** The interface is not a dashboard. It is a daily companion. Marimba greets the teacher by time of day, acknowledges the weight of the workload, and speaks in complete sentences — not bullet points or percentages. The goal is for a teacher to feel *supported*, not monitored.

**Cognitive ease over comprehensiveness.** The screen shows 3 things. Not 30. The schedule day is displayed as a quiet label — `Day 3` — not a progress bar, not a calendar widget. Information that doesn't reduce a decision is noise. Everything in the UI earns its place by reducing what the teacher has to think about.

> The hardest design constraint: nothing on screen should require explanation.

## How it works

```
Teacher's schedule + pending tasks
           |
           v
   Mistral Large 3 (AI reasoning)
           |
           v
  Top 3 prioritized actions
           |
           v
  Teacher sees 3 priority cards
           |
           v
   Tap a card → briefing + action
           |
           v
  Review, approve, mark done → next
```

The backend loads the teacher's rotating 6-day class schedule and all pending tasks (grading, communications, lesson prep). It sends this context to Mistral, which returns the 3 most relevant task IDs ranked by urgency, proximity to today's classes, and due dates. If no API key is configured, a local scoring algorithm takes over as fallback — so the app always works.

The current schedule day (1–6) is tracked with a persistent rolling calculator that correctly skips weekends and school holidays, rather than naively mapping Monday → 1. A manual override endpoint allows correcting drift caused by closures or cancelled days.

## Core User Flow

**Step 1 — Open**
Teacher opens TeacherPilot → Marimba greets them by time of day → 3 **priority cards** appear, each showing class, subject, estimated time, and urgency.

**Step 2 — Tap**
Teacher taps a card → it expands into a full briefing: attendance flags, class history, AI-suggested actions. The card itself becomes the workspace — no navigation away.

**Step 3 — Marimba handles it**
Marimba surfaces the pre-drafted output — graded rubric, email draft, lesson plan — right inside the card. Teacher reviews, tweaks if needed, and approves. Marimba reads the briefing aloud as the card opens.

**Step 4 — Voice (always available)**
Teacher speaks to Marimba at any time: *"What's urgent?"*, *"Help me prep 6B2"*, *"Mark that done."* Voxtral transcribes it, Mistral reasons about it, ElevenLabs speaks the response back. Marimba's visual state changes with what it's doing.

**Step 5 — Done**
Teacher marks the card done → card dismisses → next priority surfaces automatically → repeat until the day is handled.

> **Cards, not bullets.** Bullets were a list to read. Cards are a space to work in.

## Features

### Priority Cards (Home Screen)
Three AI-ranked cards shown at a time — no scrolling, no overwhelm. Each card displays class, subject, estimated time, and urgency. Dynamically generated by Mistral Large 3 based on today's schedule, deadlines, and pending workload.

### Marimba Voice Companion
Always-present fox avatar with four visual states (idle, listening, thinking, speaking). Voice input via Voxtral Realtime; spoken responses via ElevenLabs with a consistent voice persona. The mic button is always one tap away.

### Data Connectors

TeacherPilot pulls from the real tools the teacher already uses. No new habits required.

| Source | What lives there | Connector status |
|---|---|---|
| **Google Sheets** | Full student list (272 students, 13 groups) | Stub — in progress |
| **Google Sheets** | Student email passwords (managed by school) | Stub — in progress |
| **Toddle** | Grades, assignments, and submission status | Stub — in progress |
| **Gmail** | Parent emails, coordinator requests, communications | Stub — in progress |
| **Google Calendar** | School events, meetings, deadlines | Stub — in progress |
| **Database / Airtable / Sheets** | History of completed tasks (audit trail) | Architecture TBD |

The connector layer is designed so each source is independent — a failure in one (e.g. Toddle is slow) doesn't block the others. When a connector is unavailable, Marimba falls back to the last known data and flags it.

### Rolling Schedule Day
The current day in the 6-day rotation is calculated from a persistent state file, correctly skipping weekends and holidays. The UI shows it as a quiet `Day N` label next to today's classes. On weekends it reads `Weekend` and the empty state says *"Enjoy the rest."*

### The Classroom Card
Tapping a priority card expands it into a full briefing: class history, absence flags, and AI-generated lesson suggestions — no extra navigation required.

### Communication Triage
Email tasks open a triaged view with AI-generated summaries and urgency tags. Urgent messages get auto-drafted responses for the teacher to review and approve.

## Getting Started

### Option A — Docker (recommended for the Pi)

See the [Deploy](#deploy-raspberry-pi--cloudflare) section below.

### Option B — Local development

**Prerequisites:** Python 3.11+ and Node.js 22+

**Backend**

```bash
cd backend
cp ../.env.example ../.env   # fill in MISTRAL_API_KEY
pip install -r requirements.txt
python main.py
# → http://localhost:8000
```

**Frontend**

```bash
cd frontend
npm install
npm run dev
# → http://localhost:5173
```

No extra config needed — Vite automatically proxies `/api/*` requests to `localhost:8000`.

**API endpoints**

| Endpoint | Description |
|---|---|
| `GET /api/priorities` | Top 3 AI-ranked tasks for today |
| `GET /api/schedule` | Teacher's class schedule + current day number |
| `GET /api/schedule-day` | Current schedule day (1–6) |
| `POST /api/schedule-day` | Manual override — body: `{"date": "YYYY-MM-DD", "day": N}` |
| `POST /api/emails` | n8n webhook — receives a batch of Gmail messages for triage |
| `GET /api/health` | Health check |
| `GET /api/docs` | Interactive API docs (Swagger UI) |

> **Without a Mistral key:** priorities still work using a local scoring algorithm based on priority level, due dates, and today's schedule.

**Running tests**

```bash
cd backend
pytest tests/ -v
```

24 tests covering priority scoring, schedule filtering, Mistral integration (with fallback), and the rolling schedule day calculator.

## Deploy (Raspberry Pi + Cloudflare)

The production setup runs entirely on a home Raspberry Pi 5. A Cloudflare Tunnel exposes the app publicly without port forwarding or a static IP, routing `marimba.truvadur.com` straight to the Pi over an encrypted tunnel.

```
Internet → Cloudflare Edge → Tunnel (cloudflared) → nginx (frontend:80)
                                                          ↓
                                                   /api/* proxy
                                                          ↓
                                                  backend:8000
```

### One-time setup — Cloudflare Tunnel

1. Go to [Cloudflare Zero Trust](https://one.dash.cloudflare.com) → **Networks → Tunnels → Create a tunnel**
2. Choose **Cloudflared**, name it (e.g. `marimba-pi`)
3. On the install screen, copy just the **token** (the long string after `--token`)
4. Add a **Public Hostname**: hostname `marimba.truvadur.com` → service type `HTTP`, URL `frontend:80`
5. Save

### Deploy

```bash
# On the Pi — first time
git clone <repo> teacher-pilot-mistral
cd teacher-pilot-mistral
cp .env.example .env
nano .env   # add MISTRAL_API_KEY and CLOUDFLARE_TUNNEL_TOKEN

# Build and launch everything (backend + frontend + tunnel)
docker compose up -d --build
```

All three containers start with `restart: unless-stopped`, which means:
- If a container crashes → Docker restarts it automatically
- If the Pi reboots → Docker starts on boot and brings everything back up
- The only way containers stay stopped is if you explicitly run `docker compose stop`

### Useful commands

```bash
docker compose logs -f          # stream logs from all containers
docker compose logs backend     # backend only
docker compose restart backend  # apply a backend change without rebuilding

# Pull latest code and rebuild
git pull && docker compose up -d --build

# Local access (bypasses Cloudflare — useful for debugging on the Pi)
# http://localhost:8080
```

## Environment Variables

Copy `.env.example` to `.env` at the project root and fill in values:

```
MISTRAL_API_KEY=your-mistral-api-key-here       # Required for AI prioritization + email triage
CLOUDFLARE_TUNNEL_TOKEN=your-tunnel-token-here  # Required for production deployment
ENVIRONMENT=production                          # Optional — defaults to "development"
```

Without `MISTRAL_API_KEY` the app still works — priorities fall back to the local scoring algorithm and email triage is skipped. Without `CLOUDFLARE_TUNNEL_TOKEN` only local access is available.

## Security

- Never commit `.env` files — they are gitignored
- Use `.env.example` as the template (safe to commit, contains no secrets)
- All API keys are read from environment variables at runtime
- Report vulnerabilities responsibly before opening a public issue
